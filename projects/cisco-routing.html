Méthodologie partagée

Shared methodology

Chaque TP reprend la même séquence : nommage et sécurisation (hostname, enable secret, service password-encryption, mots de passe console et VTY), configuration des interfaces avec identification manuelle du côté DCE via show controllers Serial X/X et pose du clock rate 64000, puis activation du routage. La vérification finale combine show ip interface brief pour les états up/up, show ip route pour la table installée, et un ping multi-sauts pour valider la convergence.

Each lab follows the same sequence: naming and hardening (hostname, enable secret, service password-encryption, console and VTY passwords), interface configuration with manual DCE-side identification via show controllers Serial X/X and clock rate 64000, then routing activation. Final verification combines show ip interface brief for up/up states, show ip route for the installed table, and multi-hop ping to validate convergence.

Three routers R1, R2, R3 chained via serial links, each with a FastEthernet LAN. R3 also acts as DHCP server for the 172.31.31.0/24 network. RIPv1 enabled on all three, then tuned via passive-interface and custom timers.

R2 — RIPv1

R2(config)# router rip
R2(config-router)#  version 1
R2(config-router)#  network 192.168.13.0
R2(config-router)#  network 192.168.14.0
R2(config-router)#  network 192.168.15.0
R2(config-router)#  no auto-summary
R2(config-router)#  passive-interface FastEthernet0/0
R2(config-router)#  timers basic 5 15 15 30   ! convergence accélérée (lab)
Piège observé Observed pitfall

Les timers timers basic 5 15 15 30 accélèrent la convergence en lab mais doivent être identiques sur tous les routeurs du domaine RIP. Des valeurs divergentes produisent des boucles et des routes instables. En production, on laisse les valeurs par défaut (30 / 180 / 180 / 240).

Custom timers speed up convergence in the lab but must be identical on every router in the RIP domain. Mismatched values cause loops and unstable routes. Keep defaults (30 / 180 / 180 / 240) in production.

TP 03 — RIPv2 + VLSM

Trois routeurs, 3 interfaces Loopback sur R3 (Lo0, Lo1, Lo2) simulant des sous-réseaux distants, adressage VLSM mixant /24, /28 et /30, route de résumé 192.168.0.0/16 pointant vers Null0 sur R2 pour éviter les boucles, et redistribution de cette statique dans RIPv2.

Three routers, 3 Loopback interfaces on R3 (Lo0, Lo1, Lo2) simulating remote subnets, VLSM addressing mixing /24, /28 and /30, a summary static route 192.168.0.0/16 pointing to Null0 on R2 to prevent loops, and redistribution of that static into RIPv2.

! --- R2 : route de résumé + redistribution ---
R2(config)# ip route 192.168.0.0 255.255.0.0 Null0
R2(config)# router rip
R2(config-router)#  version 2
R2(config-router)#  network 10.0.0.0
R2(config-router)#  network 209.165.200.0
R2(config-router)#  redistribute static
R2(config-router)#  no auto-summary

! --- R3 : loopbacks pour simuler des sites distants ---
R3(config)# interface Loopback0
R3(config-if)#  ip address 172.30.200.33 255.255.255.240     ! /28
R3(config)# interface Loopback1
R3(config-if)#  ip address 172.30.200.17 255.255.255.240     ! /28
R3(config)# interface Loopback2
R3(config-if)#  ip address 172.30.110.1  255.255.255.0       ! /24
Pourquoi Null0 ? Why Null0?

Null0 est une interface poubelle : tout paquet destiné à un sous-réseau non existant de 192.168.0.0/16 est détruit au lieu d'être renvoyé. C'est ce qui permet à R2 d'annoncer le super-réseau /16 via RIPv2 sans créer de boucle sur des destinations inexistantes.

Null0 is a sink interface: any packet destined to a non-existent subnet of 192.168.0.0/16 gets dropped rather than looped back. This lets R2 advertise the /16 super-network via RIPv2 without creating loops for non-existent destinations.

Observation clé via debug ip rip

Key observation via debug ip rip

Les updates RIPv2 sont envoyées en multicast 224.0.0.9 (vs broadcast 255.255.255.255 en RIPv1), et surtout les masques réels /30, /28, /24 apparaissent dans les updates. Sans no auto-summary, ils seraient résumés à /16 aux frontières de classe — comportement inacceptable avec un adressage VLSM.

RIPv2 updates are sent as multicast 224.0.0.9 (vs RIPv1 broadcast 255.255.255.255), and crucially the actual masks /30, /28, /24 are visible in the updates. Without no auto-summary, they would collapse to /16 at class boundaries — unusable under VLSM.

! --- R2 : 4 réseaux dans area 0 ---
R2(config)# router ospf 1
R2(config-router)#  network 100.10.0.0   0.0.255.255 area 0
R2(config-router)#  network 101.10.0.0   0.0.255.255 area 0
R2(config-router)#  network 192.168.1.32  0.0.0.31   area 0   ! /27 wildcard
R2(config-router)#  network 192.168.1.128 0.0.0.31   area 0
Wildcard mask

OSPF consomme un wildcard mask, inverse du masque réseau. Pour un /27, c'est 0.0.0.31 ; pour un /16, c'est 0.0.255.255. Source d'erreur la plus classique en TP : mettre un subnet mask à la place — l'adjacence ne passe alors jamais en FULL et on perd 20 minutes à chercher.

OSPF uses wildcard masks, the inverse of subnet masks. For /27 → 0.0.0.31; for /16 → 0.0.255.255. Classic mistake: passing a subnet mask by accident — adjacency never reaches FULL and 20 minutes get burned looking for it.

Router-ID — stabilisation via Loopbackloopback-pinned

Par défaut, OSPF choisit le Router-ID en prenant la plus haute IP d'interface physique active. Résultat : on obtient des RIDs non déterministes qui changent si une interface tombe. Solution standard en production — poser une interface Loopback sur chaque routeur, prioritaire dans l'algorithme d'élection.

By default, OSPF picks the Router-ID from the highest active physical IP. Result: non-deterministic RIDs that shift when an interface goes down. Production-standard fix — add a Loopback interface on each router, which takes priority in the election algorithm. 

R1(config)# interface Loopback0
R1(config-if)#  ip address 172.16.1.1 255.255.255.0

R2(config)# interface Loopback0
R2(config-if)#  ip address 172.16.2.1 255.255.255.0

R3(config)# interface Loopback0
R3(config-if)#  ip address 172.16.3.1 255.255.255.0

R1# clear ip ospf process        ! force la relance pour appliquer le nouveau RID
Reset ALL OSPF processes? [no]: yes

Après clear ip ospf process, show ip ospf confirme les nouveaux RIDs : 172.16.1.1 / 172.16.2.1 / 172.16.3.1. Avantage réel : une interface loopback est logique, elle ne tombe jamais — le RID reste stable même en cas de panne d'interface physique.

After clear ip ospf process, show ip ospf confirms the new RIDs: 172.16.1.1 / 172.16.2.1 / 172.16.3.1. Real benefit: a loopback is logical and never goes down — the RID stays stable even under physical interface failure.

Ce que j'en retire

Takeaways

  • Toujours désactiver auto-summary sur RIPv2 — sinon VLSM cassé silencieusement.
  • Toujours fixer le Router-ID OSPF explicitement (Loopback ou router-id X.X.X.X) — les RIDs implicites bougent.
  • Toujours poser passive-interface sur les interfaces LAN — pas de voisin routeur dessus, inutile d'y envoyer des updates et d'ouvrir une surface d'attaque.
  • Premier paquet parfois perdu après convergence : c'est de l'ARP, pas un bug du protocole de routage.
  • Always disable auto-summary on RIPv2 — VLSM silently breaks otherwise.
  • Always pin the OSPF Router-ID explicitly (Loopback or router-id X.X.X.X) — implicit RIDs drift.
  • Always set passive-interface on LAN-facing interfaces — no router neighbor there, so sending updates is wasteful and opens attack surface.
  • First packet sometimes lost after convergence: that's ARP, not a routing-protocol bug.
